## 1. 背景与目标

给女友的"护理助手"App（`com.ecloudinfo.hulizhushou`，宜云健康/虚拟医院）刷题任务做自动答题。
目标App的题目正文用画布/WebView渲染，**无障碍服务读不到任何节点**（已逐窗口验证，App窗口是空的），因此**必须走「截图(takeScreenshot) + ML Kit中文OCR + dispatchGesture坐标点击」路线**，不能指望读节点拿文字/做点击。

测试机：小米 MIUI，Android 14 (API34)，分辨率 1080×2400。两台adb设备连接时必须显式 `-s 564e42ec` 指定物理机（另一台是未用的emulator-5554）。

## 2. 目标App的界面行为（已逐屏截图确认的事实）

- **任务详情页**：含"任务详情/完成条件/出题规则/错题集/开始答题"等特征词；顶部有题库**标题**（如『"三基"分册6外科护理学(烧伤+移植+神外)』）+ 一行小字**项目**（如『急诊科2026年考试』）。点"开始答题"进入做题。
- **答题页(QUESTION)**：顶部题号"N." + 类型"单选题"或"多选题"(红字，OCR常认错成"半选题"等) + 题干；选项A~E每个是「圆圈字母+文字」；底部"确定"按钮(灰=未选,粉=可点)；最底导航条 上一题|纠错上报|下一题。（题号N每轮重新计数的细节见下一条"轮/组边界"说明）
- 选中某项→圆圈变绿、确定变粉。点确定→进入反馈页。
- **反馈页(FEEDBACK)**：选错的圈变红、正确项圈变绿，并显示文字 `正确答案:X你的答案:Y`（直接是文字，OCR读取即可，不需要识别颜色）；按钮变 AI讲题|下一题。
- 一组题目第1轮固定 **15题**；**做错的题会按"轮"重复出现**（第2轮只含第1轮的错题，第3轮只含第2轮剩下的错题……），直到某一轮全部答对为止。**每一轮的题号都重新从1开始排列**，不是接着上一轮的题号继续往后数。每一轮的最后一题，反馈页按钮会从"下一题"变成"提交"。点"提交"后：这一轮**还有错题**时，**直接进入下一轮的题，不会弹出任何确认弹窗**；这一轮**全部答对**时，点"提交"会弹出"确定要提交么"弹窗（**SUBMIT_DIALOG**），点弹窗里的确定表示**整组真正完成**。即 SUBMIT_DIALOG 在一个题库里**只出现一次**（最后全对那一轮），不是每轮提交都会出现。
- 一组全对/提交完算"完成一组"，每组积分 = 题数。**错题没有惩罚**，可以放心试错来学答案。
- **题型：单选题为主，多选题确实存在，只是目前还没实测遇到过**。`ScreenModel.isMulti` 字段和 `tapSequence` 的多点位连续点击支持要保留，遇到真实多选题时要重点盯一下这条逻辑是否工作正常。

## 3. 自动答题核心规则

整体是三层，从底层往上：**盲选（核心）→ 暴力模式（盲选 + 顺便OCR建题库）→ 智能模式 / 自动化（在暴力预选之上用题库答案替换 + 整组循环刷分）**。实现顺序也按这个来：先盲选，再暴力，最后自动化。

三者的关系：每次作答都先跑一遍暴力盲选逻辑算出一个"预选答案"，再看"智能模式flag"是否开启——开了就把预选答案替换成题库里查到的该题答案（查不到/不可信则保留盲选预选）。所以**智能模式 = 暴力流程 + 一层"查表替换"，盲选是两种模式共同的地基**。

### 3.1 盲选——核心底层（两种模式都依赖）

- **轮换节奏**：没学到答案时，盲选当前"轮"锁定的字母——第1轮所有未知题统一选 **A**，进入下一轮统一改 **B**，再下一轮 **C**……（对选项数取模循环）。轮/组边界的界面事实见第2节（最后一题按钮变"提交"后，这一轮没全对就直接进下一轮、不弹窗；全对才弹出 SUBMIT_DIALOG，确认 = 组真正完成）。
   - 举例：第1次进入任务，第1轮=15题（题号1~15）；第15题（本轮最后一题）按钮变"提交"→点提交→假设这15题里有5道错的→**没有弹窗**，直接进入第2轮，题号重新从1数到5；第2轮第5题按钮变"提交"→点提交→假设又错2道→**没有弹窗**，直接进入第3轮，题号从1数到2；……直到某一轮（比如第4轮）全部答对，那一轮点"提交"才第一次弹出 SUBMIT_DIALOG，确认后才是整组真正完成。
   - **盲选字母推进的唯一触发点 = 上面的"轮"边界**（每轮最后一题按钮变"提交"且点了提交），不是按题、也不是按其它任何事件推进。
   - **（✅已修复，见6.5）** 触发点 = "反馈页动作按钮=提交"这个每轮都出现的事件（点提交→进下一轮时 `blindRound++`），由 `ScreenParser.isSubmit` 识别。早先错接在 SUBMIT_DIALOG 弹窗(只在组末出现一次)导致字母不换、卡死，已改正。题号从大变小只能做辅助(题号可能一直是1)。
- **有限轮内必中（仅单选题）**：每轮统一换一个字母、选项最多5个，最多5轮盲选必然命中一次。这是**暴力模式**的收敛保证；**智能模式**因为第一次错后就从反馈页学到答案、下一轮直接命中，通常1轮就够。⚠️ 此保证**不适用多选题**（单字母盲选凑不出多字母答案，见5.5）。
- **不能永久卡死在同一题上**：即便OCR识别、坐标计算出了任何没预料到的差错，也必须有兜底能跳出死循环，而不是无限重复同一题。已实测出现过两类卡死：① 第一轮提交后进入第二轮错题、题号重排时反复卡在同一题；② 题库识别错误时一直卡住。
   - **无法自愈时的处理**（✅已实现，见6.6）：连各种兜底/自救都无法恢复(长时间停在 UNKNOWN)时，不死等不无限重试，而是：① 提示用户；② 按 3.3 回收临时资源；③ 停止、停在"等用户手动点开始"。阈值=连续5次UNKNOWN(可设)。

### 3.2 暴力模式——盲选 + 顺便OCR建题库

- 作答完全靠 3.1 的盲选，**不读题库**来决定点哪个选项。
- 但**照常做OCR**（识别题干 + 反馈页的"正确答案:X"），把学到的"题→答案"**写进题库**。因此暴力模式跑几遍就能把一个题库的答案攒出来，可以先用暴力走几遍、再切智能模式秒答。
- **答案记忆按题库分开存**：题库标识 =「任务详情页」的 标题 + 项目，两者都相同才算同一题库；切换题库时先存旧库再载入新库；每个题库存一个独立文件（含标题/项目/更新时间/题→答案），方便以后单独删除过期题库（已在 `MainActivity` 题库管理界面里做了"删除"按钮）。
   - **（✅已修，见6.4）** 早先"标题+项目完全相等"判断过严，OCR 噪声会把同一题库拆成多个文件；现已改为 `bankId` 归一化 + 0.84 模糊匹配复用同库，并自动合并已拆开的重复库。

### 3.3 智能模式 / 自动化——在暴力之上叠加

- **智能模式 = 暴力预选 + 查表替换**：先按 3.2/3.1 算出盲选预选答案，再检查"智能模式flag"是否开启；开了就用题库查到的该题答案**替换**预选（**有题库答案就用，不管第几轮**；只有查不到的题才落到盲选）；查不到、或该存档答案连续判错被标记不可信（`failCount`/`distrusted`）则退回盲选预选。
- **查表的范围与性能**：`get()` **只查当前这一个题库**（`setBank` 已把匹配到的库载进内存，不跨库搜，所以题库个数/文件数不影响速度、也不会串答案）。查一次 = 先 O(1) 精确命中(题干稳定时是常态)，没命中再对**当前库**逐题算 Levenshtein 相似度容忍 OCR 噪声——所以慢只跟"当前库题目数"线性相关，几十~几百题是亚毫秒级，相对步长可忽略。
- **选错时的学习（覆盖）规则**（两种模式都生效，因为都写库）：反馈页的"正确答案"是绝对权威。当我们选了某个答案、反馈页却显示错了时，用 `yourIdx`（反馈页"你的答案"，即实际选中项）和我们这一题"打算点的下标"对比来决定怎么处理：
   - `yourIdx == 打算点的下标` 且 `yourIdx != correctIdx` → 说明**答案本身错了**（不是手滑），用 correctIdx **覆盖**题库里这条，下次这题再出现就秒对。
   - `yourIdx != 打算点的下标` → 说明**点击没点中**（点偏/被遮挡），不是答案错 → **不覆盖题库**，避免把一条本来正确的答案改坏；这次按"点击失败"处理。
   - 同一题连续判错到阈值（`failCount`/`distrusted`，见第4节）→ 多半是"两道不同题被去重归到了同一个 key"之类的结构问题，覆盖也救不了（会来回横跳）→ 停止信任这个 key、退回纯盲选，保证不卡死。
   - 这套"区分答案错/没点中"的判断，正是 6.3"点下一题前再识图校验"要支撑的目的。
- **整组循环**：一组做完（提交+确认）后自动回到任务详情页，自动点"开始答题"进入下一组；做到设定的目标组数后自动停止。目标组数是**用户可改的设置项**，默认 `targetGroups = 14`，由用户按需修改即可，不需要去对任务页显示的分数（那个分数口径不用管）。
- **临时文件/资源要尽量自动回收，不能越用越大**。下面把"产生哪些、存哪、保留还是删、何时删"列清楚（均为代码实测确认）：

   **所有文件都在 App 的外部私有目录下**：`getExternalFilesDir(null)`，即 `/Android/data/com.quizhelper.dumptool/files/`，App 卸载时系统会整体清掉。其下只有两个子目录：

   | 目录/文件 | 由谁产生 | 内容 | 处置 |
   |---|---|---|---|
   | `banks/bank_<hash>_<len>.json` | `AnswerStore.persist()`，每次 `put()` 学到答案就落盘 | 每个题库一个JSON：`{title, project, updated, answers:{题干→正确字母}}` | **永久保留**（学习成果）。只能由用户在 `MainActivity` 题库管理界面**手动**逐个删除，自动回收**绝不**碰它 |
   | `dumps/step_*.txt` `dumps/ocr_*.txt` `dumps/node_*.txt` | `saveToFile()`，**仅**调试广播 STEP/OCR/NODES 触发 | OCR/节点/逐帧的纯文本导出（调试取证用） | **临时**，自动回收对象。注意：**自动答题本身不写 `dumps/`**，只有人手动发调试广播才会写 |

   - **不会产生截图文件**：目前没有任何截图(.png)被写到磁盘。截图只是内存里的 `Bitmap`，OCR 完即弃（见下条）；`dumps/` 里只是几个 KB 级的文本。
   - **内存资源（待办）**：每次 `captureScreen` 拿到的 `Bitmap`（1080×2400 ARGB_8888 ≈ 10MB）用完后**没有显式 `recycle()`**，靠 GC 回收。高频截图循环下这是真正更值得关心的"资源"。建议在 OCR 回调结束后显式 `bmp.recycle()`，列为待办（不属于"删文件"，但属于"回收资源"）。
   - **`clearTemp()` 的语义**：只删 `dumps/` 目录下的文件，**不动** `banks/`。`DumpAccessibilityService.clearDumps()` 直接委托给它。
   - **自动触发删除的时机**（已落地，见第4节）：
      - 每轮提交确认(`SUBMIT_DIALOG`)
      - `stopAuto()`（手动停止 / 达到目标组数自动停止）
      - 服务 `onDestroy`/`onUnbind`（系统关闭或解绑无障碍服务，走 `cleanup()`）
      - 进程崩溃前的未捕获异常兜底(`installCrashCleanup`，尽力而为——Android 不保证崩溃时还能跑代码)
      - 以上都不依赖 `MainActivity` 那个手动"清理临时文件"按钮（它只是额外的兜底入口）
   - **dev机器上的** `logcat_latest.txt` 是调试时手动重定向出来的，不在 App 关心范围内，已加进 `.gitignore`。

### 3.4 运行流程——一个 step 的完整生命周期

自动答题是一个**定时循环**：每隔一个 step 间隔（按当前模式取值，暴力默认700ms / 智能默认1300ms，见6.7）跑一个 step，一个 step = "截一次图 → OCR → 判断当前是什么页面 → 按页面类型做一次动作 → 排下一个 step"。代码入口 `scheduleStep()` → `loopStep()`，循环只在 `auto==true` 时继续。**每个 step 恰好截 1 次图、做 1 次 OCR**（不是连续不停地截）。

一个 step 内部按顺序：

1. **隐藏悬浮条**：`setOverlayVisible(false)`。必须先把我们自己的悬浮控制条藏掉，否则它会被截进图里、甚至我们的点击会落到悬浮条上。等 120ms 让它真正消失。
2. **截图**：`takeScreenshot(DEFAULT_DISPLAY)` 拿到 `HardwareBuffer` → 转成 `Bitmap`(ARGB_8888) → `buffer.close()`。截图失败则本 step 直接结束、排下一个。
3. **OCR**：`OcrEngine.recognize(bmp)`（ML Kit 离线中文）出文字+每行包围盒。
4. **纠错 + 解析**：每行先过 `OcrFix.fix()` 纠错表（见6.1），再 `ScreenParser.parse()` 解析成 `ScreenModel`——判定页面类型 `kind`（QUESTION/FEEDBACK/TASK_DETAIL/SUBMIT_DIALOG/UNKNOWN）并算出要点的坐标、正确答案/你的答案下标等。
5. **按页面类型动作** `act(model)`：
   - **TASK_DETAIL**（任务详情页）：若 `groupInProgress` 为真（说明刚从做题状态回到这页）→ 判定**一组完成**：`groupsDone++`、`blindRound=0`、清空 `failCount`，达到目标组数就 `stopAuto()`。然后 `setBank(标题,项目)` 切到对应题库，`groupInProgress=true`，点"开始答题"。
   - **QUESTION**（答题页）：保存题干到 `lastQuestionText`；按 3.1/3.3 算出要点的下标（智能命中→存档答案；否则/不可信→`blindRound` 盲选字母）；`tapSequence` 依次点选项（多选用）再点"确定"。
   - **FEEDBACK**（反馈页）：把识别到的"正确答案"`store.put()` 写进题库（用 `lastQuestionText` 当 key，因为反馈页题干末尾会多出解析文字）；用 `yourIdx` 核实命中/失败、维护 `failCount`（见3.3、第4节）；点"下一题"。
   - **SUBMIT_DIALOG**（提交确认弹窗，只在整组全对那一轮出现）：点"确定"完成整组，`clearDumps()`。（`blindRound++` 已挪到 FEEDBACK 的"提交"分支，这里不再 ++，见3.1/6.5。）
   - **UNKNOWN**（✅已加兜底，见6.6）：先尝试点掉可关闭按钮(`dismissBtn`)自救；连续到 `Prefs.unknownLimit`(默认5)仍 UNKNOWN 则 `stopAuto` 提示用户、停下等人工。不再无限死等。
6. **收尾**：`finishStep()` → `setOverlayVisible(true)` 恢复悬浮条 → `scheduleStep()` 排下一个 step。

> 注：`STEP`/`OCR`/`NODES` 是另一套**手动调试广播**，只截图+解析+把结果写进 `dumps/` 文本，不点击、不参与上面的自动循环。

## 6. 已决定要做、待实现的方案

> 不再单设"待澄清问题"章节：有疑问当场沟通确认；未实现的方案统一记在本节，每条把"已定的"和"未定的"分开写清。

### 6.1 OCR 等价对照表——改成"等价组"+ 可加载 JSON（✅表已落地，自动学习=建议待用户确认，未实现）

**核心改动（✅已落地）：从"错→对"改成"互相等价"。** 早先 `OcrFix` 是 `(错文本→对文本)` 的有向硬编码列表。改成**等价组**：每组内的写法互相可互换、组内第一个当规范代表，`fix()` 把组内其它写法都替换成代表。**好处**：绕开"哪个才是对的"这种判不准的难题（OCR 对某些字总是认错，"多数票"会学反），只为把同一文字的变体归一成同一个，让题库 key、按钮匹配稳定。
- **可加载 JSON（✅已落地）**：表不再硬编码，放 `assets/ocrfix.json`（随 APK 发布的初始全表）+ `files/ocrfix_user.json`（用户确认新增的，同格式，叠加在内置表之上）。`OcrFix.load(context)` 在服务启动时载入；读不到退回内置 FALLBACK。格式 `{"groups":[["代表","变体1","变体2"], ...]}`。
- **初始表（✅已落地，可继续扩）**：`assets/ocrfix.json` 已按"尽量全"原则种了一批——UI词(单选题/多选题/下一题/正确答案…)、真机实测的医学词混淆(股骨/殷骨、骨膜/骨腹、破坏/玻坏、幻肢/幻胶、病人/病入…)、题库名变体(急诊科/急诊利/怠诊科、考试/孝试)。**优先用词级(多字)分组，少用单字**，避免把正常文本里的字误替换。

**自动学习（✅已实现：建议+确认，绝不自动生效）**：`OcrLearn` 对象。
- **攒候选**：`AnswerStore.canonical` 把两段题干判成"同一题"(模糊命中、非全等)时调 `OcrLearn.observe(a,b)`，逐位比对(同长度、差异≤3字)，把每个差异字位记成一条无向候选等价对并计数，持久化到 `files/ocrfix_candidates.json`(每20次/停止时落盘)。
- **展示门槛**：`qualifying()` 返回**总观察 ≥30(`MIN_TOTAL`) 且 该对占 X、Y 各自全部混淆比例都 ≥97%(`MIN_DOMINANCE`，即是干净的成对混淆而非乱混)**的候选，按次数降序。
- **方向(优先显示对的可能性大的)**：`OcrLearn.noteText` 统计每个字在题干/题库名里的全局出现次数(`charFreq`)；展示时把**字频高的那个当代表放前面**——正确字在每次正确识别里都出现(频率高)、错字只偶尔出现(频率低)，所以频率高的更可能对。确认入表时代表(第一个)就是规范写法。题库名的字差异也会喂进来(见6.4)。
- **UI**：`MainActivity` "OCR 等价对候选(待确认)"区，每条**带一句上下文例句**(取该字位附近窗口，让用户看懂在替换什么)，显示 `例：…急诊科2026年考试…` + `把「孝」当成「考」（更可能对的是「考」）· n次` + **确认等价**(写入 `ocrfix_user.json`、`OcrFix.load` 立即生效)/**忽略**两个按钮。永久行为完全由用户点确认，阈值只决定"要不要拿出来给你看"。

### 6.2 暴力 / 智能两种模式——同一套底层 + 一个UI开关切换（✅已实现）
核心架构见第3节（3.1盲选 / 3.2暴力 / 3.3智能）。这里只记实现要点。**关键修正**：旧版本写的"暴力模式完全不读不写题库"已作废——暴力模式照常OCR、照常写题库，只是作答不查表。两种模式只差"作答时是否用题库答案替换盲选预选"这一层。

- **暴力模式（flag关）**：每道未知题按 3.1 盲选作答（不查题库决定点哪个）；但仍OCR、把反馈页学到的答案写进题库。胜在简单、不被题干模糊匹配出错拖累；代价是同一题库下次重复出现也不会"秒对"，要多跑几轮。适合先用它把一个新题库的答案攒出来。
- **智能模式（flag开）**：在暴力盲选预选的基础上，用题库查到的该题答案替换预选；查不到/连续判错不可信(`failCount`/`distrusted`)则保留盲选。
- 两种模式共享同一套"轮/组边界"和 `blindRound` 兜底机制（3.1），区别只在 3.3 那层"查表替换"。

**默认模式（已定）：默认暴力模式**，可在设置页勾选切换到智能。理由与推荐用法：新题库还空时，智能模式查不到答案、跟暴力没区别还多一层查表，所以**推荐先用暴力跑几遍把答案攒进题库，攒得差不多了再切智能秒答**。两种模式区别见上面两条（暴力=纯盲选作答+照样建库；智能=暴力预选之上用题库答案替换）。

**开关位置（已确认）**：**不放悬浮控制条**，放在 `MainActivity` 的设置页面里（和题库管理列表同一个Activity）。因为开关在 `MainActivity`、实际生效在 `DumpAccessibilityService`，两者是不同组件，需要一个持久化的中间层（如 `SharedPreferences`）传递状态，而不能像悬浮条按钮那样直接改服务内存里的字段。

**待实现**：
- `MainActivity` 设置页加一个模式开关（如 `Switch`/`CheckBox`），状态写入 `SharedPreferences`（建议复用 `AnswerStore` 所在的偏好文件或新开一个专用文件）。同一个设置页再加几个输入框：**目标组数**（默认14，替代硬编码 `targetGroups`）、**暴力速度 / 智能速度两个 step 间隔**（默认 700 / 1300，见6.7）、**UNKNOWN 连续次数阈值**（默认5，见6.6），都用户可改、存同一份 `SharedPreferences`。
- `DumpAccessibilityService` 每次需要判断时从同一个 `SharedPreferences` 读取当前值（或在 `onSharedPreferenceChangeListener` 里缓存，避免每帧都读文件）——这样运行期间在 `MainActivity` 切换，正在跑的服务下一次截图循环就能读到新值并立即生效。
- `act()` 的 `QUESTION` 分支：先算盲选预选；若 flag=智能，再用 `store.get()/keyFor()` 查表替换（含 `failCount`/`distrusted` 不可信回退）。`FEEDBACK` 分支：两种模式都跑 `store.put()` 把学到的答案写进题库（这是和旧设计最大的不同），同时跑 yourIdx/correctIdx 校验日志。
- 开关读写的 `SharedPreferences` 默认值设为"暴力"。

### 6.3 选中核实——只做"事后用 yourIdx 判断"，方案B(提交前取色)不做（✅已定）
目的：确认"实际选中的答案 == 我们预设/打算点的答案"，让 3.3 的覆盖学习能分清"答案错"还是"没点中"。

**结论**：
- **只用事后判断（已实现，免费）**：反馈页本来就显示"你的答案"，`ScreenParser` 解析成 `yourIdx`，3.3 用它区分点中/没点中、决定是否写库。无需额外截图/取色。
- **方案B（提交前主动取色复核）不做**：错题被允许且量很少，为省"偶尔点偏错一题"却要在每道题点确定前多一次截图+取色，太拖速度，不划算。
- 方案A(纯OCR复核反馈页文字)也不做。

### 6.4 题库同一性改模糊匹配 + 合并重复库（✅已实现）
现状bug(已修)：同一题库被 OCR 噪声拆成多个文件。真机实测一个"分册5"题库被拆成 **4** 个、"分册6"拆成 2 个，变的全是噪声：标题开头 `|"三基"` / `"三基"` / `三基"`(漏引号)、项目 `急诊科`↔`急诊利`、`考试`↔`孝试`。

**关键数据(决定阈值的依据)**：把标题+项目归一化(只留中文/数字/字母)后算相似度——
- 同一题库的噪声变体之间 ≈ **0.92**(只差1~3个字)；
- 两个不同真题库之间 ≈ **0.73**(靠括号内容"骨科+心胸+泌尿"vs"烧伤+移植+神外"和分册号5/6区分，差异大)。
- 两者分得很开，取 **0.84** 居中阈值即可干净区分。

**已落地实现**(`AnswerStore`)：
- `bankId(title,project)` = (标题+项目)去标点/引号/竖线/括号/加号，只留中文数字字母(分册号、括号内容、科室、年份这些区分信息都保留)。
- `setBank` 不再用"完全相等"，改成：和当前库 `bankId` 相似度 ≥ `bankSim`(0.84) 就算同库不切换；否则在所有已存题库里找相似度 ≥0.84 的复用。
- **顺手去重**：若找到多个相似库(就是被拆开的重复)，复用最大的那个作目标，把其余库的答案 `putAll` 并进来、`delete` 多余文件，合并结果落盘——既防新重复，也清理已有的，不丢已学答案。
- 文件名从 `bankId` 的 hash 生成；`currentKey` 改为直接持有 `currentFile`。
- **显示名取最干净的形(不让错的显示在上面)**：合并时各变体过 `OcrFix.fix`(纠急诊利→急诊科、孝试→考试…)+去掉竖线，取多数(并列取最长)作 `currentTitle/currentProject` 落盘；`MainActivity` 列表也对名字再过一遍 `OcrFix.fix`(兼容老库)。题库名变体之间的字差异(去标点后逐位比)还会喂给 6.1 的 OCR 候选，让"考/孝、科/利"这类名字误识别也进待确认列表。

**仍以 6.8 为人工兜底**：万一阈值误并/漏并，用户可在题库管理里改/删。阈值 0.84 偏保守(宁可让噪声变体合并，也不轻易把两个真库并一起——真库间 0.73 远低于 0.84)。

### 6.5 blindRound 推进信号——接到"提交按钮"，不是接弹窗（✅已实现）
先把事实理清（这是两种不同的"提交"）：

1. **每一轮的边界 = 反馈页动作按钮显示"提交"**（每轮最后一题，按钮从"下一题"变成"提交"）。点这个"提交"按钮：
   - 这一轮还有错题 → **直接进入下一轮**（不弹窗）。
   - 这一轮全部答对 → **才弹出"确定要提交"确认框（SUBMIT_DIALOG）**，确认后整组完成。
2. 所以"整组完成 = 弹窗"，把**组完成检测**绑在 SUBMIT_DIALOG 弹窗上是对的。
3. **盲选字母推进（blindRound++）该接在"反馈页动作按钮=提交"这个事件上**（每轮都会出现），而不是接在弹窗上。

**当前代码的真实毛病**：`blindRound++` 现在写在 `SUBMIT_DIALOG` 分支里，而弹窗只在"整组最后全对那一轮"才出现一次——所以组内各轮之间 `blindRound` 根本不推进，盲选字母一直不换，未知题答案若不等于当前字母就会一轮轮一直错、卡死。**触发点接错了对象**（接到了只在组末出现的弹窗，应接到每轮都有的"提交"按钮）。这仍是"卡死循环"的重点怀疑根因。

**修复方向**：
- `ScreenParser` 在反馈页要能**区分动作按钮是"提交"还是"下一题"**（现在 `nextBtn` 不分二者），多一个标志位（如 `isSubmit`）。
- `FEEDBACK` 分支点的若是"提交"按钮（=本轮最后一题，即将进下一轮）→ `blindRound++`。
- `SUBMIT_DIALOG` 分支保留点确定（组完成检测），那里的 `blindRound++` 可去掉或保留无所谓（组完成时 `blindRound` 会清零）。
- **题号从大变小**可作为**辅助**判据，但**不能单独用**：题号可能"一直是1"（某轮只剩最后一题时，题号就是1，看不出变化）。所以以"提交按钮"为主信号，题号变化顶多做佐证。

**结论（已定）**：把 `blindRound++` 从弹窗(SUBMIT_DIALOG)挪到"反馈页那个'提交'按钮被点时"。剩下的只是实现细节（ScreenParser 加 `isSubmit` 标志位区分提交/下一题按钮）。

### 6.6 UNKNOWN无法自愈时——提示用户+回收资源+停止（✅已实现）
行为：检测到"自己解决不了" → 提示用户 → 按第3.3节清理临时文件 → 停止自动循环 → 等待用户手动重新点击开始，而不是无限等待或无限重试。

**已确认的判定与处理**：
- 两段是**串联**关系，不是二选一：每帧 UNKNOWN 先按 5.2 的"通用自救"尝试（点"确定/关闭/X/我知道了"或返回键）；**连续 5 次 UNKNOWN 仍没恢复**（即自救也没用）才判定无法自愈、走停止。
- **阈值默认 5 次，做成用户可设置**（放 6.2 那个 `MainActivity` 设置页，存同一份 `SharedPreferences`，默认5）。
- 触发后**提示 + 停止 + 等待用户操作**。已实现：toast + **系统通知栏 Notification**(`notifyStop`，无人值守时 toast 一闪而过看不到，Notification 留在通知栏)。非手动停止(达标完成/卡住)都会发通知；Android13+ 首次进 MainActivity 申请 `POST_NOTIFICATIONS`，没授权也不影响主流程(通知静默丢弃)。

**实现备注**：题库学到的答案是即时落盘的，停止时不用额外处理；停止后由用户手动重新点"开始"继续。

### 6.7 step 间隔用户可调，且暴力/智能两个模式分开设速度（✅已实现）

**一个 step 的实测耗时（2026-06-16 真机日志，stepInterval=1300ms 时）**：
- QUESTION→FEEDBACK ≈ 2.32s；FEEDBACK→QUESTION ≈ 1.56s；**答完一道题(Q步+F步) ≈ 3.9s**（14题/54s = 3.86s/题，吻合）。
- 拆解一道题的 3.9s：`2×stepInterval`(1300×2=**2.6s，大头**) + tapSequence选项→确定(350+400≈0.75s) + 2×(截图前等120ms + OCR≈150ms)≈0.55s。
- 即：**主要时间被 `2×stepInterval` 吃掉**，压 stepInterval 收益最直接。

**两个模式分开设速度（已定）**：
- **暴力模式**不看题干文字、只需认出页面类型 + 选项坐标，且答错无惩罚，**容忍更快/更糙的截图**——可以用更小的间隔。
- **智能模式**要靠干净 OCR 去题库匹配，需要更稳的间隔，太快会因残影/动画中间态导致题干 OCR 抖动、匹配失败。
- 因此 `Prefs` 存**两个**间隔：`stepIntervalBruteMs`（默认 700）、`stepIntervalSmartMs`（默认 1300）。`scheduleStep` 按当前模式取对应值。两个都用户可调（设置页两个输入框），范围都 300~5000ms。

**估算**：brute@700 一道题 ≈ 2×700+0.75+0.55 ≈ **2.9s/题**（比 1300 的 3.9s 快约 25%）；想更快可继续往下压(到 400~500)，由用户按手机流畅度自调。

**暂不做**：把"截图前固定等待 120ms"也暴露成可调——先只调主间隔，够用再说。

### 6.8 题库可展开查看 + 手动修正——也是"匹配把握不足"的人工兜底（✅已实现查看/改答案/删除；合并/拆分可选项未做）

**为什么要做**：6.1（题干去重/OCR自学）和6.4（题库同一性）本质是同一个"求吻合度"问题，都复用 `AnswerStore.canonical()`/`similarity()`（归一化后 Levenshtein 相似度）。但纯算法的"把握"天生不牢：阈值高了漏并（同题被拆成两条/两库），低了误并（两题被并成一条、答案互相覆盖横跳）。案例组合题、OCR 噪声都会制造例外。**与其追求算法 100% 置信，不如让用户能看、能改**——算法尽力且取保守（宁漏并不误并），剩下的错由用户顺手修，改完即 ground truth。这才是最务实的"把握"来源。

**要做**：`MainActivity` 题库管理界面，每个题库**点击可展开**，列出它下面所有"题干 → 已学答案"条目，用户能：
- **查看**：扫一眼有没有明显错的（答案被 OCR 读错、两题被误并成一条、同题漏并成两条）。
- **修正**：改某条的答案、删除某条；（可选）把两条手动合并、或把误并的一条拆开。
- 改动直接落盘到该题库 JSON，下次答题直接命中修正后的值。

**实现要点**：`AnswerStore` 已有 `listBanks()`、每库 JSON 里就是 `answers:{题干→字母}`，读出来展开即可；需要新增"编辑某条/删除某条/写回 JSON"的接口。题库文件结构不用变。

**配套（写进 6.1/6.4 的匹配策略）**：
- 匹配阈值取**保守**（偏向漏并），因为误并比漏并更难被用户发现、危害更大（答案互相覆盖）；漏并最多是题库多几条、靠 6.8 合并即可。
- **把"正确答案是否一致"作为文字相似度之外的独立佐证**：文字相似且答案相同→更敢判同题；文字相似但答案不同→很可能是两题或识别错，宁可不并。
