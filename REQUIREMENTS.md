## 1. 背景与目标

给女友的"护理助手"App（`com.ecloudinfo.hulizhushou`，宜云健康/虚拟医院）刷题任务做自动答题。
目标App的题目正文用画布/WebView渲染，**无障碍服务读不到任何节点**（已逐窗口验证，App窗口是空的），因此**必须走「截图(takeScreenshot) + ML Kit中文OCR + dispatchGesture坐标点击」路线**，不能指望读节点拿文字/做点击。

测试机：小米 MIUI，Android 14 (API34)，分辨率 1080×2400。两台adb设备连接时必须显式 `-s 564e42ec` 指定物理机（另一台是未用的emulator-5554）。

## 2. 目标App的界面行为（已逐屏截图确认的事实）

- **任务详情页**：含"任务详情/完成条件/出题规则/错题集/开始答题"等特征词；顶部有题库**标题**（如『"三基"分册6外科护理学(烧伤+移植+神外)』）+ 一行小字**项目**（如『急诊科2026年考试』）。点"开始答题"进入做题。
- **答题页(QUESTION)**：顶部题号"N." + 类型"单选题"或"多选题"(红字，OCR常认错成"半选题"等) + 题干；选项A~E每个是「圆圈字母+文字」；底部"确定"按钮(灰=未选,粉=可点)；最底导航条 上一题|纠错上报|下一题。（题号N每轮重新计数的细节见下一条"轮/组边界"说明）
- 选中某项→圆圈变绿、确定变粉。点确定→进入反馈页。
- **反馈页(FEEDBACK)**：选错的圈变红、正确项圈变绿，并显示文字 `正确答案:X你的答案:Y`（直接是文字，OCR读取即可，不需要识别颜色）；按钮变 AI讲题|下一题。
- 一组题目第1轮固定 **15题**；**做错的题会按"轮"重复出现**（第2轮只含第1轮的错题，第3轮只含第2轮剩下的错题……），直到某一轮全部答对为止。**每一轮的题号都重新从1开始排列**，不是接着上一轮的题号继续往后数。每一轮的最后一题，反馈页按钮会从"下一题"变成"提交"。点"提交"后：这一轮**还有错题**时，**直接进入下一轮的题，不会弹出任何确认弹窗**；这一轮**全部答对**时，点"提交"会弹出"确定要提交么"弹窗（**SUBMIT_DIALOG**），点弹窗里的确定表示**整组真正完成**。即 SUBMIT_DIALOG 在一个题库里**只出现一次**（最后全对那一轮），不是每轮提交都会出现。
- 一组全对/提交完算"完成一组"，每组积分 = 题数。**错题没有惩罚**，可以放心试错来学答案。
- **题型：单选题为主，多选题确实存在，只是目前还没实测遇到过**。`ScreenModel.isMulti` 字段和 `tapSequence` 的多点位连续点击支持要保留，遇到真实多选题时要重点盯一下这条逻辑是否工作正常。

## 3. 自动答题核心规则

整体是三层，从底层往上：**盲选（核心）→ 暴力模式（盲选 + 顺便OCR建题库）→ 智能模式 / 自动化（在暴力预选之上用题库答案替换 + 整组循环刷分）**。实现顺序也按这个来：先盲选，再暴力，最后自动化。

三者的关系：每次作答都先跑一遍暴力盲选逻辑算出一个"预选答案"，再看"智能模式flag"是否开启——开了就把预选答案替换成题库里查到的该题答案（查不到/不可信则保留盲选预选）。所以**智能模式 = 暴力流程 + 一层"查表替换"，盲选是两种模式共同的地基**。

### 3.1 盲选——核心底层（两种模式都依赖）

- **轮换节奏**：没学到答案时，盲选当前"轮"锁定的字母——第1轮所有未知题统一选 **A**，进入下一轮统一改 **B**，再下一轮 **C**……（对选项数取模循环）。轮/组边界的界面事实见第2节（最后一题按钮变"提交"后，这一轮没全对就直接进下一轮、不弹窗；全对才弹出 SUBMIT_DIALOG，确认 = 组真正完成）。
   - 举例：第1次进入任务，第1轮=15题（题号1~15）；第15题（本轮最后一题）按钮变"提交"→点提交→假设这15题里有5道错的→**没有弹窗**，直接进入第2轮，题号重新从1数到5；第2轮第5题按钮变"提交"→点提交→假设又错2道→**没有弹窗**，直接进入第3轮，题号从1数到2；……直到某一轮（比如第4轮）全部答对，那一轮点"提交"才第一次弹出 SUBMIT_DIALOG，确认后才是整组真正完成。
   - **盲选字母推进的唯一触发点 = 上面的"轮"边界**（每轮最后一题按钮变"提交"且点了提交），不是按题、也不是按其它任何事件推进。
   - **⚠️ 已落地代码里的一个设计错误（待修复方案见7.5）**：目前编译验证过的实现是把 `blindRound++` 放在 `SUBMIT_DIALOG` 分支里。但 SUBMIT_DIALOG 在一个题库里几乎只会在**最后**触发一次（组完成时），不会在组内的轮次之间触发——这意味着 `blindRound` 在一个题库做完之前**几乎不会变化**：如果某道未知题的正确答案不是当前锁定的字母，它就会一轮又一轮一直错、永远等不到换字母的机会。**这很可能是下面"卡死循环"现象的真正根因之一**，比之前怀疑的"未识别弹窗"更基础、优先级更高。新的轮边界信号需要换成一个每轮真正结束时都会触发的事件（候选方向：检测题号从大变小，即"重新从1开始"），具体方案待讨论，**不要在未确认前直接改代码**。
- **有限轮内必中**：每轮统一换一个字母、选项最多5个，最多5轮盲选必然命中正确答案一次（命中后该题答案被反馈页学到、存进题库）。
- **不能永久卡死在同一题上**：即便OCR识别、坐标计算出了任何没预料到的差错，也必须有兜底能跳出死循环，而不是无限重复同一题。已实测出现过两类卡死：① 第一轮提交后进入第二轮错题、题号重排时反复卡在同一题；② 题库识别错误时一直卡住。
   - **无法自愈时的处理**：如果连各种兜底/自救尝试都无法让程序自己恢复正常（比如长时间停在 UNKNOWN，怎么点都恢复不了），不应继续死等或无限重试，而是要：① 提示用户"卡住了，需要人工介入"；② 按 3.3 的机制回收本次运行的临时资源；③ 停止自动循环，停在"等待用户手动点开始"的状态。具体"判定为无法自愈"的阈值、提示方式待讨论，见 7.6。

### 3.2 暴力模式——盲选 + 顺便OCR建题库

- 作答完全靠 3.1 的盲选，**不读题库**来决定点哪个选项。
- 但**照常做OCR**（识别题干 + 反馈页的"正确答案:X"），把学到的"题→答案"**写进题库**。因此暴力模式跑几遍就能把一个题库的答案攒出来，用户可以先用暴力走几遍、再切智能模式秒答。（注：这一点与旧版7.2"暴力模式完全不读不写题库"的描述相反，以这里为准；7.2 待据此更新。）
- **答案记忆按题库分开存**：题库标识 =「任务详情页」的 标题 + 项目，两者都相同才算同一题库；切换题库时先存旧库再载入新库；每个题库存一个独立文件（含标题/项目/更新时间/题→答案），方便以后单独删除过期题库（已在 `MainActivity` 题库管理界面里做了"删除"按钮）。
   - **新发现的bug（待修复，方案见7.4）**：这个"标题+项目完全相等"的判断过严——实测同一个真实题库被拆成了两个题库文件，学到的答案没法共享。需要改成更鲁棒的匹配方式（关键片段提取或模糊匹配），具体算法待讨论确认，不要在没确认方案前就动手改。

### 3.3 智能模式 / 自动化——在暴力之上叠加

- **智能模式 = 暴力预选 + 查表替换**：先按 3.2/3.1 算出盲选预选答案，再检查"智能模式flag"是否开启；开了就用题库查到的该题答案**替换**预选；查不到、或该存档答案连续判错被标记不可信（`failCount`/`distrusted`）则退回盲选预选。
- **选错时的学习（覆盖）规则**（两种模式都生效，因为都写库）：反馈页的"正确答案"是绝对权威。当我们选了某个答案、反馈页却显示错了时，用 `yourIdx`（反馈页"你的答案"，即实际选中项）和我们这一题"打算点的下标"对比来决定怎么处理：
   - `yourIdx == 打算点的下标` 且 `yourIdx != correctIdx` → 说明**答案本身错了**（不是手滑），用 correctIdx **覆盖**题库里这条，下次这题再出现就秒对。
   - `yourIdx != 打算点的下标` → 说明**点击没点中**（点偏/被遮挡），不是答案错 → **不覆盖题库**，避免把一条本来正确的答案改坏；这次按"点击失败"处理。
   - 同一题连续判错到阈值（`failCount`/`distrusted`，见第4节）→ 多半是"两道不同题被去重归到了同一个 key"之类的结构问题，覆盖也救不了（会来回横跳）→ 停止信任这个 key、退回纯盲选，保证不卡死。
   - 这套"区分答案错/没点中"的判断，正是 7.3"点下一题前再识图校验"要支撑的目的。
- **整组循环**：一组做完（提交+确认）后自动回到任务详情页，自动点"开始答题"进入下一组；做到设定的目标组数后自动停止。**当前代码默认目标 `targetGroups = 14`（对应210分）**，但测试任务页显示的分数曾从690到封顶2470，**目标组数/分数的确切口径还没最终确认**（见第6节）。
- **临时文件/资源要尽量自动回收，不能越用越大**：
   - 适用范围：调试导出(STEP/OCR/NODES，目前存在 `dumps/` 目录下)，以及后续任何新增的临时截图/诊断文件（比如第5节第2点提到的"把UNKNOWN画面存一份方便适配"，一旦做了同样要纳入回收）。
   - **不适用**：`banks/` 下的题库答案JSON——那是要长期保留学习成果的，不能被当成临时文件清掉。
   - 要求覆盖的清理时机：每组完成、用户手动停止、达到目标组数自动停止、无障碍服务被系统关闭/解绑(`onDestroy`/`onUnbind`)、进程异常崩溃（尽力而为——Android 不保证崩溃时还能跑代码，但至少注册一个未捕获异常兜底，崩溃前抓紧清一次）。不能只靠 `MainActivity` 里那个手动"清理临时文件"按钮。

## 4. 已落地的兜底/健壮性机制（本次会话新增，已编译验证）

- **选项坐标行合并阈值修复**（`ScreenParser.clusterRows` 合并阈值 50px→65px）：根因是选项文字换行后多出的单字残行（如"血"字单独成行）离主选项行约51px，超过原50px阈值被误判成独立"选项行"，把后面所有选项坐标顶歪一位，导致"答案查对了但点的位置是错的"。已用真实卡死案例（高灌注综合征题）截图日志定位并验证修复（修复后该题第一次点击就直接命中真实坐标）。
- **"你的答案"核实机制**（`ScreenParser` 新增 `yourIdx` 字段，解析反馈页 `正确答案:X你的答案:Y` 里的Y部分）：之前系统只看自己「打算点哪个下标」来判断对错，这次改成看反馈页真实显示的"你的答案"，这样才能查出"我们以为点对了，但实际点击没命中"这类问题。
- **已存答案连续失败计数 + 自动降级为盲选**（`failCount` / `distrusted` / `failLimit=2`）：如果某题用存档答案点击后，反馈页显示的"你的答案"连续2次都不等于"正确答案"，就认为这个存档答案不可信，之后改走 `blindRound` 轮流盲选（保证能跳出死循环），不再无限信任可能有问题的存档/坐标。`failCount` 在开始新一轮自动答题、或每组完成时清空。
- **临时文件自动回收**（对应第3.3节）：`clearDumps()` 现在直接复用 `AnswerStore.clearTemp()`（去重），并在四个地方都会触发——① 每轮提交确认(`SUBMIT_DIALOG`，原有)；② `stopAuto()`（手动停止 / 达到目标组数自动停止，新增）；③ 服务 `onDestroy`/`onUnbind`（系统关闭或解绑无障碍服务，新增）；④ 新增全局未捕获异常兜底(`installCrashCleanup`)，进程崩溃前尽力清一次再把异常交还系统默认处理（不吞异常，只是顺手清理）。`banks/` 题库答案文件不受影响，只清 `dumps/` 调试目录。

## 5. 当前已知缺口（待处理，优先级从高到低）

1. **（新发现，最高优先级）blindRound 推进信号绑定到了一个几乎不会在轮间触发的事件**：见第3.1节详细说明——已落地代码把 `blindRound++` 放在 `SUBMIT_DIALOG` 分支，但 SUBMIT_DIALOG 实际只在题库最后全对那一轮触发一次，中间轮次的"提交"根本不弹窗。结果是 `blindRound` 在一个题库做完之前几乎不会变化，未知题如果答案不是当前锁定字母就会一直错、永远等不到换字母——**很可能是用户反复反馈的"卡死循环"的真正根因**，比下面第2点的"未识别弹窗"更基础。修复方向（待讨论，见7.5）：换成检测"题号从大变小"作为新的轮边界信号。
2. **未识别的弹窗/插页导致死等**：`ScreenParser` 只认 QUESTION/FEEDBACK/TASK_DETAIL/SUBMIT_DIALOG 四种正常页面，碰到其它弹窗会落到 `UNKNOWN`，而 `act()` 的 `UNKNOWN` 分支只打日志"等待"，不会做任何尝试，本次会话实测在这个状态卡了近7分钟。
   - **2026-06-16 用户已纠正**：当时弹出的就是已经能识别的"确定要提交"确认框(SUBMIT_DIALOG)本身——**不是**之前猜测的"一个全新、未识别的组完成弹窗"。所以根因还需重新定位，目前怀疑方向：① 点 `dialogConfirm` 坐标没有真正点中/生效，弹窗一直没消失，但某些帧OCR碰巧没能稳定匹配到"确定要提交"/"所有题目已作答"关键词（噪声/遮挡），于是这一帧被分类成 `UNKNOWN`，下一帧又识别回 SUBMIT_DIALOG，反复横跳但弹窗从未真正被关掉；② 还有别的、目前没想到的原因；③ 现在看也可能和上面第1点有关——如果当时其实是中间轮次（不该有弹窗），那个"弹窗"可能本来就是别的东西，需要结合第1点的修复重新观察。
   - 取证还没拿到：本次会话曾用 `STEP` 想抓这个弹窗的OCR原始内容，但抓到的时候弹窗已经消失/翻页过去了（抓到的是正常的FEEDBACK页）。**下次复现，第一时间发 `STEP` 广播**（不要等，不要先做别的），连续抓两三次（间隔1~2秒）把这期间每一帧的OCR原始行列表都存下来，才能看出弹窗到底有没有被点掉、点击坐标和弹窗实际按钮位置是否吻合。
   - 通用兜底（即便取证前也可以先做）：连续N次 `UNKNOWN` 后尝试点截图里能找到的"确定/关闭/X/我知道了"类按钮，或者退一步用"返回"键，作为"不认识但先尝试自救"的兜底，避免无限期等待。**2026-06-16 用户补充**：如果连这些自救尝试都不能让画面恢复正常，下一步应该是提示用户+回收资源+停止（见第3.1节"无法自愈时的处理"、第7.6节），而不是无限重试下去。
3. **（新发现）题库匹配过严，同一题库被误判成两个**：见第3.2节"答案记忆按题库分开存"——实测同一个真实题库被拆成两个题库文件。修复方向（待讨论，见7.4）：用标题/项目里的稳定关键片段或模糊匹配代替"完全相等"判断。
4. **目标组数/分数口径不确定**：默认 `targetGroups=14`（210分），但用户提过测试任务页分数显示从690到封顶2470，需要用户确认到底按"组数"还是"App内任务的分数门槛"来停止，以及具体数值。
5. **任务详情页标题行可能丢字**：`ScreenParser` 用 `cy in 200..410` 配合"含数字/单选/多选/选题/判断"做表头行识别，已发现至少一例"单选题"和题干第一行被OCR粘在同一行从而整行被当表头排除、丢失题干开头几个字（如"病人,男,65岁。高血压及糖尿病"）的情况；目前靠 fuzzy-match 容忍，但严格来说题干文本不完整，长期可能影响存取key的稳定性，未正式修复。

### 7.1 OCR纠错替换表——增加"自动学习"，置信度≥95%才能入表
现状：`app/src/main/java/com/quizhelper/dumptool/OcrFix.kt` 是一个 `(错误文本, 正确文本)` 的硬编码列表，`toLines()` 对每一行OCR结果都会过一遍（如 `"苔案"→"答案"`、`"ー"→"一"`），目前只能手动加行。

**用户决定**：可以做成自动学习，但要求"百分之95的把握以上才能添加"。

**置信度怎么算——我的提案，待你确认**：复用 `AnswerStore` 已有的模糊匹配基础设施（Levenshtein 相似度，阈值0.82）。当两段文本被判定为"同一道题"（相似度达标但不是完全相同）时，回溯它们之间具体差在哪几个字符，就是一条"候选替换对"（比如这次差在"苔"/"答"）。把这种候选对在**整个使用过程**（不分题库）里出现的次数和方向一致的比例都记下来（持久化成一个新的小文件，类似 `AnswerStore` 的JSON存法）：
- 样本量达到下限（比如 ≥5 次观察）**且**方向一致比例 ≥95%（即几乎每次出现这种差异，都是同一个方向的替换，没有出现过反方向的情况）时，才自动把这条 `(错误文本→正确文本)` 写入持久化的"学得替换表"，之后自动套用到所有后续OCR结果，和手工表一起生效。
- 没达标的候选对只是先记着次数，不生效，不影响现有逻辑。
- 学习信号在暴力和智能两种模式下都会产生（两种模式都做OCR、都建题库，见3.2/3.3）。**待厘清**：OCR纠错的"候选替换对"来自"两段文本被判为同一道题但有字符差异"的比较，而这种比较发生在"拿当前题去题库里查已有key"的时刻；暴力模式作答时不查表，但建题库写入时是否也对现有key做一次模糊匹配（以避免重复条目）会决定它能否产生纠错学习信号——这个细节要在确定3.2写库实现方式时一并定下来。

### 7.2 暴力 / 智能两种模式——同一套底层 + 一个UI开关切换
核心架构见第3节（3.1盲选 / 3.2暴力 / 3.3智能）。这里只记实现要点。**关键修正**：旧版本写的"暴力模式完全不读不写题库"已作废——暴力模式照常OCR、照常写题库，只是作答不查表。两种模式只差"作答时是否用题库答案替换盲选预选"这一层。

- **暴力模式（flag关）**：每道未知题按 3.1 盲选作答（不查题库决定点哪个）；但仍OCR、把反馈页学到的答案写进题库。胜在简单、不被题干模糊匹配出错拖累；代价是同一题库下次重复出现也不会"秒对"，要多跑几轮。适合先用它把一个新题库的答案攒出来。
- **智能模式（flag开）**：在暴力盲选预选的基础上，用题库查到的该题答案替换预选；查不到/连续判错不可信(`failCount`/`distrusted`)则保留盲选。

**开关位置（已确认）**：**不放悬浮控制条**，放在 `MainActivity` 的设置页面里（和题库管理列表同一个Activity）。因为开关在 `MainActivity`、实际生效在 `DumpAccessibilityService`，两者是不同组件，需要一个持久化的中间层（如 `SharedPreferences`）传递状态，而不能像悬浮条按钮那样直接改服务内存里的字段。默认状态待定（见下）。
