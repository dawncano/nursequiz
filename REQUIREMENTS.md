## 1. 背景与目标

目标 App：护理助手（`com.ecloudinfo.hulizhushou`，宜云健康/虚拟医院）。本工具帮女友自动完成其中的**刷题、视频挂课、考试**任务。

**技术路线：无障碍控件树直读 + 坐标点击。**
- **读屏**：`NodeParser.toModel(targetRoot())` 直接读目标 App 的无障碍节点树，解析成 `ScreenModel`（页面类型 + 选项/按钮坐标 + 正确答案/你的答案下标等）。零识别错误、不漏选项、不怕 `FLAG_SECURE`，不需要任何纠错表/几何外推。
- **操作**：`dispatchGesture` 按节点 `bounds` 中心坐标点击。
- **入口**：`loopStep()` 按 `Prefs.mode()` 把每一拍分给对应状态机——答题 `QuizMachine` / 悬浮答案 `FloatMachine` / 视频 `VideoMachine` / 考试 `ExamMachine`（详见 3.6）。
- **关键配置**：`accessibility_service_config.xml` 的 `accessibilityFlags` **必须含 `flagIncludeNotImportantViews`**。护理助手的题干/选项/确定/反馈正确答案都是原生 `TextView`，但未标记 important-for-accessibility——默认无障碍服务看不到，加上该 flag 后才逐字可读（uiautomator 能读到也是因为它默认带这个 flag）。

**功能范围**：自动答题（一边刷一边建库）· 悬浮答案模式（只显示答案、自己点）· 视频自动挂课 · 考试模式。四者在设置页互斥切换（见 5.2）。

**隐蔽性**：无障碍服务名伪装成 `com.google.android.accessibility.selecttospeak.SelectToSpeakService`（避按服务名检测，见 7.1）；持有 `WRITE_SECURE_SETTINGS` 后可自助开启无障碍（`A11yEnabler`），免去手动进系统设置开。

**测试机**：小米 MIUI，Android 14（API 34），分辨率 1080×2400。两台 adb 设备同时连接时须显式 `-s 564e42ec` 指定物理机（另一台为未使用的 emulator-5554）。

## 2. 目标 App 的界面行为（已逐屏截图确认）

> 本节只记录**目标 App 长什么样**（界面事实），与我们怎么读怎么点（见 3、5 节）无关。

### 2.1 答题任务

- **任务详情页**：含"任务详情/完成条件/出题规则/错题集/开始答题"等特征词；顶部有题库**标题**（如『"三基"分册6外科护理学(烧伤+移植+神外)』）+ 一行小字**项目**（如『急诊科2026年考试』）。点"开始答题"进入作答。
- **答题页（QUESTION）**：顶部题号"N." + 类型"单选题"/"多选题"（红字）+ 题干；选项 A~E 每个是「圆圈字母+文字」；底部"确定"按钮（灰=未选，粉=可点）；最底导航条 上一题|纠错上报|下一题。选中某项 → 圆圈变绿、确定变粉。点确定 → 进入反馈页。
- **反馈页（FEEDBACK）**：选错的圈变红、正确项圈变绿，并显示文字节点 `正确答案:X你的答案:Y`（是文字节点，直读即可，不需识别颜色）；按钮变 AI讲题|下一题。
- **轮次规则**：一组第 1 轮固定 **15 题**；**做错的题按"轮"重复出现**（第 2 轮只含第 1 轮错题，依此类推），**每轮题号都重新从 1 开始**，直到某轮全部答对。每轮最后一题反馈页按钮从"下一题"变"提交"。点"提交"后：
  - 这轮**还有错题** → 直接进下一轮，**不弹确认窗**。
  - 这轮**全部答对** → 弹"确定要提交么"窗（**SUBMIT_DIALOG**），点确定 = **整组真正完成**。
  - 即 SUBMIT_DIALOG 一个题库**只出现一次**（最后全对那轮），不是每轮提交都弹。
- 一组全对/提交完算"完成一组"，每组积分 = 题数。**错题没惩罚**，可放心试错攒答案。
- **题型**：单选为主；多选确实存在但尚未真机遇到；判断题若是 A.对/B.错 走同一"圈字母+文字"结构。`ScreenModel.isMulti` + `tapSequence` 多点连点支持须保留，遇真实多选题重点验证。

### 2.2 视频任务（2026-06-29 真机抓取确认）

视频模块同样能读无障碍节点；播放器是 **Google ExoPlayer**（节点 id 带 `exo_content_frame`/`exo_overlay` 等）。

- **视频任务详情页**（结构同答题任务详情）：标题、`进行中`、`完成条件：视频进度达到95%`、`观看进度 X%`（**独立叶**，如 `观看进度 0%`）、`累计时长 00:26:28`、`学习人数`；下方视频列表，每行 = 序号+名称（**详情页是连体单叶** `1. 8.1自我管理`，名称**不含 .mp4**——2026-06-29 旧记录的 `xxx.mp4` 是误记）+ 时间叶 `已学 / 总长`（详情页连体单叶 `00:00 / 10:57`）+ 进度条；底部 `开始观看` 按钮。
  - ⚠️ **`观看进度 X%` 必须只从"观看进度"那一叶取**——页面另有静态文案 `完成条件：视频进度达到95%`，若取全页最大% 会把 95 当成已看进度→一进详情即误停（2026-07-03 真机实测的"秒停"根因，已修 `NodeParser.toVideoModel`）。
  - **播放器页列表结构与详情页不同**（2026-07-03 真机 dump）：每行序号/名称/时刻**全部拆成独立叶**——`1. `（纯序号叶）+ `8.1自我管理`（名称叶）+ `10:57`（总长，右上叶）+ `04:12`（累计已学，右下叶）。`toVideoModel` 现以"纯序号叶 `^\d+\.$`"为锚（名称 `8.1…` 也形如数字点，用"点后无内容"区分），右侧两时刻按 top 排 first=总长/last=累计，行 tap = 屏幕水平中心×行 Y。
- **播放器页**（点开始观看后**页内**展开，非独立 Activity）：`exo_content_frame` 视频面；中间大 `▶` 覆盖层**仅暂停时存在**（播放时消失）；底部控件条 = 播放/暂停键 + **当前位置** `00:00` + 进度条 + **总时长** `19:22` + 倍速 `1.0x` + 清晰度 `640P` + 全屏。
- **播放/暂停判定**：读"当前位置/累计时长"是否在增长——增长=播放中、冻住=暂停（最可靠）；辅助看中间 `▶` 覆盖层是否存在。单点视频中心只切换控件显隐、不切播放态。
- **完成度量（2026-07-03 二轮校准，重要）**：完成条件=**整体观看进度 ≥95%**，但该 % 是**缓存死值**——播放器页播放中不更新、详情页要手动刷新 ↻ 才更新，**播放中不可用来判完成**。列表「**累计时长**」更是 replay 累加会超总长、**完全不能**判完成。**唯一可靠且实时的完成信号 = 每条视频的绿色进度条填充比**（=唯一覆盖率）：进度条无文字，用 `allRects`+`barCoverage()` 按几何(左≈88 宽 View 簇、填充右/容器右)读出。全条填满 ⇒ 整体≥95%。

### 2.3 考试任务（含反作弊机制，2026-07-02 逆向 base.apk 确认）

竞品逻辑：考试**无逐题反馈**（交卷后才出成绩），题目页结构大概率同练习；用"交卷/考生/答题卡/剩余时间"等标志区分考试/练习。确切的按钮文案（确定/下一题/交卷）、交卷确认窗层级、考试标志文字仍待一场真实考试 dump 节点校准。

**反作弊机制（原版 `apk_analysis/apks/base.apk` 反编译结论，Hermes 字节码 v94 反汇编产物位于 `apk_analysis/decompiled/`）：** 护理助手是 React Native + Hermes 应用；**考试页是一个 WebView（H5）**。反无障碍检测链路：

1. **进场触发点**：考试 WebView 的 **`onLoadEnd`（H5 加载完那一刻）** 调用 RN 的 `checkAccessibilityService()`——这是"进场查一次"的检测点（对应界面上的入场签到→进考试页）。
2. **RN 只上报、不判罚**：`checkAccessibilityService` 调原生模块 `AccessibilityInfoModule.getEnabledAccessibilityServices()` 拿到 `{isEnabled, services}`，然后 `webview.postMessage({type:'accessibilityServiceResult', isEnabled, services, ...})` **把结果丢进 H5**。真正"拦不拦"的判罚逻辑在**远端 H5 里**（不在 APK bundle 内，反不到）。
3. **复查是事件驱动、非定时轮询**：RN 侧 `checkAccessibilityService` 的全部调用点 = `onLoadEnd`（进场）+ H5 通过 `onMessage` 发来的 `type=='accessibilityServiceClosed'` 时（"提示你去关无障碍→你关完回来"后 H5 主动请求复查）。**RN 侧没有 `setInterval` 定时轮询**。H5 消息还有 `EXAM_STARTED`（`handleExamStarted`）正式开考等。
4. **破解版怎么过的**：破解版 `护理助手'7.6.9.APK` 的 bundle 把第 2 步那个 `getEnabledAccessibilityServices` 上报调用删了 → H5 永远收到 `isEnabled:false` → 直接放行（另 `classes3.dex` 带 `SignatureKiller843` 做签名绕过以便重打包）。

**对"错峰"方案的意义**：进场检测（onLoadEnd）能骗过——**进场前关掉无障碍**，上报 isEnabled:false，H5 放行即可。**唯一残余命门**：开考后再开无障碍，会不会被 H5 自己的复查抓到——**H5 是否在考中轮询，APK 里看不到**，只能真机进一次考试、开着无障碍观察它会不会突然弹"检测到无障碍"。据此设计的考试模式见 5.8。

## 3. 自动答题核心规则

**【2026-06-25 简化：去掉"暴力/智能"双模式，统一为单一行为（参考竞品）。】**
每题统一流程：**控件读题 → 查当前题库（命中就按存档答案点，没命中就盲选）→ 点确定 → 反馈页读"正确答案"一律写库 → 下一题**。即"一边刷题一边建库"，第一遍主要靠盲选攒库、之后同题直接命中。盲选仍是没命中时的地基（见 3.1）；查库失败时还会先试一次 AI 兜底（`AiHook`，当前空实现返回 null→自然盲选）。不再有模式开关、不再有暴力/智能两套速度。

### 3.1 盲选——核心底层

- **恒选 A**：未命中题库的题统一盲选 **A**，不再按"轮"换字母。控件直读 + 反馈页 100% 可靠，错题靠"反馈页学到正确答案 → 下次重现时查库命中"自愈（≤2 次必对），旧的 A→B→C 轮换对冲已无意义、连同 `blindRound` 字段及其全部触发点一并删除（见 5.2）。故也不再识别/计数"轮"。
- **多选题**：单字母盲选凑不出多字母答案，但每题都查库——第 1 轮盲选错 → 反馈页学到完整答案（如 ABD）→ 第 2 轮查表命中，自然不卡死。
- **不会永久卡死在同一题**：连续达到 `Prefs.unknownLimit`（默认 5，用户可调）次 UNKNOWN 仍无法自愈时，提示用户、回收临时资源、停止循环，等待人工介入。

> 第 1 轮固定 15 题、错题按"轮"重复、SUBMIT_DIALOG 只在整组全对那轮出现等**目标 App 的界面行为**见 2.；那是 App 的出题规则，与我们这边"盲选恒 A"的实现无关。

### 3.2 查库与建库（统一作答逻辑）

- **每题都查库**：`skipBank = distrusted`（只有存档答案被标记不可信才跳过查库）。命中 → 按存档答案点；没命中 → 盲选（3.1）。控件直读零误差，查库即时，没有"为了快而不查表"的理由，故不再有暴力模式。
- **照常建库**：反馈页"正确答案"一律写进题库——不管这题是查库答的还是盲选答的。第一遍遇到的新题主要靠盲选，反馈页学到正确答案入库，下次同题直接命中。
- **答案记忆按题库分开存**：题库标识 = 任务详情页的标题 + 项目，归一化后取 0.84 模糊匹配，实质相同才算同一题库；切换题库时先存旧库再载入新库；每个题库存一个独立文件，方便单独删除过期题库。
- **多选题**：单字母盲选凑不出多字母答案，但因为现在每题都查库，多选题第 1 轮盲选错 → 学到完整答案（如 ABD）→ 第 2 轮查表命中，自然不卡死（不再需要"多选破例查表"的特判）。

### 3.3 自动化——整组循环

- **查表范围**：`get()` 只查当前一个题库（`setBank` 已把匹配到的库载进内存）。精确命中 O(1)，未命中则对当前库逐题算 Levenshtein 相似度——题库百题量级为亚毫秒级，可忽略。
- **选错时的学习（覆盖）规则**：反馈页"正确答案"是绝对权威。通过 `yourIdx`（反馈页"你的答案"）与打算点的下标比对来决定处理方式：
  - `yourIdx == 打算点的下标` 且 `yourIdx != correctIdx` → **答案本身错了**，用 correctIdx **覆盖**题库。
  - `yourIdx != 打算点的下标` → **点击未命中**（点偏/被遮挡），不覆盖题库，避免把正确答案改坏。
  - 同一题连续判错到阈值（`failLimit=2`）→ 停止信任该 key，退回纯盲选，保证不卡死。
- **整组循环**：一组完成（提交+确认）后自动回到任务详情页，自动点"开始答题"进入下一组；达到目标组数（`targetGroups`，默认 14，用户可调）后自动停止。

### 3.4 临时文件与资源管理

所有文件位于 App 外部私有目录 `getExternalFilesDir(null)`（`/Android/data/com.quizhelper.dumptool/files/`），卸载时系统整体清除。

| 目录/文件 | 产生来源 | 内容 | 处置 |
|---|---|---|---|
| `banks/bank_<hash>_<len>.json` | `AnswerStore.persist()`，每次 `put()` 触发 | 每个题库一个 JSON：`{title, project, updated, answers:{题干→正确字母}}` | **永久保留**（学习成果）。只能由用户在 `MainActivity` 题库管理界面手动删除，自动回收绝不碰此目录 |
| `dumps/node_*.txt` `dumps/nodeq_*.txt` | `saveToFile()`，仅调试广播 NODES/NODEQ 触发 | 节点树 / 题目解析的纯文本导出（调试取证用） | **临时**，自动回收。自动答题本身不写 `dumps/`，只有手动发调试广播才写 |
| `unhandled/unhandled_*.txt` | `captureIfUnhandledQuestion()`，遇疑似未适配题型自动 dump | 未识别画面的节点树（待照真实结构补适配，见 5.1） | **半永久**：独立目录、**不被 `clearTemp` 清**，留作事后取证；只随"清空所有数据"一并删 |

- **不截图**：控件树直读不产生任何图片/Bitmap（OCR 截图整摊已删，见 5.9），故无"截图落盘/回收"一说。
- **`clearTemp()` 语义**：只删 `dumps/` 目录，不动 `banks/`、`unhandled/`。
- **自动触发删除的时机**：SUBMIT_DIALOG 提交确认、`stopAuto()`、服务 `onDestroy`/`onUnbind`、未捕获异常兜底（`installCrashCleanup`，尽力而为）。不依赖 `MainActivity` 手动"清理临时文件"按钮（它只是额外兜底入口）。
- **彻底清空**：`MainActivity` 提供"清空所有数据"按钮（确认后 `getExternalFilesDir` 整目录删——题库 / `dumps` / `unhandled` 全清 + 重置所有设置）；`android:allowBackup=false` 禁止云备份还原。
- 后续若新增临时文件，须写进 `dumps/`（或同样纳入 `clearTemp` 范围），保持"临时文件统一回收"。
- 节点调试导出（NODES/NODEQ）UI 无按钮，只能用 `adb broadcast` 触发。

### 3.5 悬浮球操作说明

悬浮球是自动答题的唯一控制入口，常驻屏幕上方，可随时拖动。

**三级形态**（单击逐级展开，闲置自动收回）：

| 形态 | 外观 | 触发条件 |
|---|---|---|
| **球态**（默认） | 半透明 D 形半球，贴左/右边缘，圆弧朝外 | 闲置超过 `attachDelayMs`（默认 3000ms）自动收回；开始/继续点后立即收回 |
| **胶囊态** | 展开显示当前组数（如 `5/14`） | 单击球态 |
| **完整态** | 显示操作按钮 | 单击胶囊态 |

**完整态按钮（按运行状态变化）**：

| 运行状态 | 显示内容 |
|---|---|
| 空闲 | 开始（`0/14`） |
| 运行中 | 暂停 \| 结束 |
| 暂停中 | 继续（`N/14`） \| 结束 |

暂停/结束操作后完整态停留数秒再收回（留时间确认操作）；开始/继续点后立即收回成球。

**拖拽与吸边**：直接拖动球体任意位置，松手后自动吸附到左边缘或右边缘（取最近侧）；球态贴边时圆弧方向随左/右自动切换。

**拖到底部关闭**：拖拽时屏幕底部浮出 ✕ 靶区；将球拖压到 ✕ 上并松手 → 停止自动答题 + 移除悬浮球（无障碍服务保留）。重新打开 `MainActivity` 后悬浮球自动唤出（`onResume` 发送 `SHOW` 广播）。

**与点击的关系**：控件树直读不截图，且悬浮球在屏幕边缘、点击都落在屏幕中部，二者不重叠，故答题主循环**不再每步藏球**（藏球是 OCR 截图时代的遗留，已删，见 5.9）；`setVisible` 仅暂停/停止时用。

**相关设置**（`Prefs` / `MainActivity` 设置页）：`attachDelayMs`（闲置收回延迟，默认 3000ms）。

### 3.6 运行流程——一个 step 的完整生命周期

自动答题是一个**循环**：每一拍跑一个 step = "读一次控件树 → 判断当前页面 → 按页面类型做一次动作 → 等下一屏就绪再排下一拍"。代码入口 `scheduleStep()`（初始踢一脚）/ `scheduleNextStep()`（步间 waitFor）→ `loopStep()`，循环只在 `auto==true && !paused` 时继续。**全程不截图、不做 OCR**——直接读无障碍节点树。

一个 step 内部按顺序：

1. **模式分派**：`loopStep()` 按 `Prefs.mode()`（优先级 视频>考试>悬浮>普通，互斥见 5.2）分给对应状态机——视频 `VideoMachine`、考试 `ExamMachine`、悬浮答案 `FloatMachine`、普通答题 `QuizMachine`（各状态机分层见 5.9）。
2. **读题**：`NodeParser.toModel(targetRoot())` 从目标 App 窗口根节点直读出 `ScreenModel`（页面类型 `kind`：QUESTION/FEEDBACK/TASK_DETAIL/SUBMIT_DIALOG/UNKNOWN + 选项/确定/正确答案/你的答案下标等，坐标取自节点 `bounds` 中心）。读不到则等下一屏。
3. **记屏签名**：`lastStepSig = sigOf(model)`，供 waitFor 判断"屏变没变"。
4. **按页面类型动作** `act(model)`：
   - **TASK_DETAIL**：若 `groupInProgress` 为真（刚从做题状态回到此页）→ 判定一组完成：`groupsDone++`、清空 `failCount`，达到目标组数就 `stopAuto()`；否则 `setBank(标题,项目)` 切库、`groupInProgress=true`、点"开始答题"。组计数只在"进入本页首帧"结算（`firstVisit`，防过渡屏重复计数，见 4）。
   - **QUESTION**：保存题干到 `lastQuestionText`；算出要点的下标（查库命中→存档答案；否则→盲选 A，见 3.1）；`tapSequence` 依次点选项再点"确定"。
   - **FEEDBACK**：把读到的"正确答案"`store.put()` 写进题库（用 `lastQuestionText` 当 key）；用 `yourIdx` 核实命中/失败、维护 `failCount`；点"下一题"或"提交"。
   - **SUBMIT_DIALOG**（只在整组全对那一轮出现）：点"确定"完成整组，进入提交冷却（见 4）。
   - **UNKNOWN**：先尝试点可关闭按钮（`dismissBtn`）自救，并把疑似未适配题型 dump 到 `unhandled/`（见 5.1）；连续达到 `Prefs.unknownLimit` 仍 UNKNOWN 则提示用户+停止。
5. **收尾**：`finishStep()` → `scheduleNextStep()`（waitFor 等下一屏就绪，拟人开则再补随机延时，见 5.4）排下一拍。

> `NODES`/`NODEQ` 是另一套**手动调试广播**，只读节点+解析+把结果写进 `dumps/` 文本，不点击、不参与自动循环。

### 5.1 题型覆盖与未适配题型自动捕获

- **已覆盖**：单选 / 多选 / 判断——只要选项是"圈字母 A–E + 文字"结构，`NodeParser` 都按选项路径解析（判断题若是 A.对/B.错 也走这条）。多选第 1 轮盲选错 → 反馈页学到完整答案（如 ABD）→ 第 2 轮查表命中。多选尚未真机实测，`isMulti` 待复验。
- **未适配（待真机看到结构再精确补，不靠猜）**：填空/简答（竞品做法 `setText("。。。")` 占位提交）、图片选项题、长题选项滚出屏外需滚动后再点。三基练习是否真有这些题型未知。
- **自动捕获探针 `captureIfUnhandledQuestion()`**：答题循环遇到判成 UNKNOWN 的画面，自动把节点树 dump 到 `files/unhandled/unhandled_*.txt`（独立目录、不被 `clearTemp` 清、同屏去重）。重新跑一遍后任何未适配题型会自动留下真实结构，照之精确补适配——避免"改前没看"的老坑。

### 5.2 作答逻辑与两种输出模式

**统一作答**：每题先查库（`store.get`，模糊匹配题干，见 5.3）→ 未命中调 `AiHook`（当前空桩恒返回 null）→ 仍没有则**盲选 A**（见 3.1）。反馈页一律读"正确答案"写库（`store.put`），一边刷一边建库。

**自愈**：覆盖规则用 `yourIdx`（反馈页"你的答案"）与"打算点的下标"比对——相等且 ≠ 正确 → 答案本身错、用正确答案**覆盖**题库；不等 → 点击没命中（点偏/被遮挡），不写库免把好答案改坏。存档答案连续判错到 `failLimit=2` / 下标超界 / 同题卡住多帧 → 标记不可信（`distrusted`）退回盲选 A，反馈页再覆盖自愈，保证不卡死。

**两种输出模式（`Prefs.floatMode`，设置页开关，默认关）**：
- **自动点击（默认）**：自动点选项 + 确定 + 下一题，整组循环（走 `QuizMachine`，见 3.6）。
- **悬浮答案（手动，`FloatMachine`）**：不点任何东西，只在一枚悬浮小标签显示**正确答案原文**（选项文字，如 `胃底静脉曲张`，多选空格分隔；查不到显示 `unknown`，非答题页占位 `。。。`），让用户自己点。此模式**不显大球**，用**手机【音量+】键**开始/结束（`onKeyEvent` 拦 `KEYCODE_VOLUME_UP`，竞品同款隐蔽触发；需 `flagRequestFilterKeyEvents`；注：`adb input keyevent` 注入的键不经无障碍过滤器，只能真机硬件键验证）。反馈页照常建库。

**模式互斥**：悬浮答案 / 视频 / 考试 三开关在设置页**互斥**（`onModeToggle`：开一个关另两个 + 刷新勾选）；运行期 `Prefs.mode()` 再按优先级 视频>考试>悬浮>普通 解析成单一模式，双保险（防两开关同开时考试抢点）。

### 5.3 题库同一性——模糊匹配 + 合并重复库

`AnswerStore`（相似度/编辑距离/归一化算法已抽到纯函数 `TextMatch`，见 5.9）：

- `bankId(title,project)` = 标题+项目去标点/引号/竖线/括号/加号，只留中文数字字母（`TextMatch.normalize`）。
- `setBank` 用模糊匹配（相似度 ≥0.84，`TextMatch.similarity`）复用同库；若发现多个相似库（读题文本的标点/细微差异拆出的重复），复用最大的那个，其余答案 `putAll` 并入、删多余文件、合并落盘。显示名取各变体清洗后的多数（并列取最长）。
- **题干 key 归一化**：`get`/`put` 先 `TextMatch.normalize` 去标点空白，精确命中走 O(1) 快路径；否则对当前库逐题算 Levenshtein 相似度（≥0.82 视为同题）。案例组合题共享很长病史前缀、只末尾子问题不同，额外卡末尾 40 字的后缀门槛，防不同子题误并。
- 阈值取保守（宁漏并不误并）：同库噪声变体 ≈0.92、不同真题库 ≈0.73，bankSim=0.84 / simThreshold=0.82 居中干净区分。万一误并/漏并，可在题库管理界面（5.6）手动改/删兜底。

真机实测一个"分册5"题库被拆成 4 个文件、"分册6"拆成 2 个，现已可识别并自动合并。

### 5.4 节奏控制——waitFor 等屏就绪 + 拟人开关

步与步之间不傻等固定毫秒，而是**等下一屏真正就绪**再走，并叠一层可选的拟人随机延时：

- **waitFor（屏就绪）**：动作收尾走 `scheduleNextStep` → `pollForReady`，每 90ms 轮询当前屏的轻量签名 `sigOf`（类型+题干+标题），**当"和上一步操作的屏不同(changed) 且 与上一拍相同(stable，避开动画中途)"或到 1.6s 兜底**就进下一拍。更快（屏加载快就走得快，真机确定→反馈 0.34s、下一题→读题 0.42s，不再被固定 700ms 拖住）、更稳（根治"读太早"竞态）。与提交冷却（见 4）正交：waitFor 管"屏就绪没"，冷却管"反馈页闪回别重复提交"。
- **拟人开关（`Prefs.humanize`，默认开）**，叠在 waitFor 之上：开=屏就绪后再 `postDelayed` 一段随机延时（800~1500ms/步）模拟人手节奏、防风控（真机 ON≈2.2s/题；竞品 2~2.5s）；关=飞速，屏一就绪立刻走（真机≈1.66s/题）。随机延时只在"屏变了"这条加，超时兜底不加，免拖慢卡死检测。
- **`Prefs.stepIntervalMs`（默认 700）退居内部**：已不是用户面的"速度"设置（由拟人开关收编，设置页不再有"单步毫秒"框），仅供 `tapSequence`（多选连点 gap=`iv/3`、选完到确定 settle=`iv/2`）等少数固定延时按比例缩放，及 `startAuto`/`resume` 初始踢一脚。

### 5.5 UNKNOWN 自救 + 无法自愈时停止

UNKNOWN 帧先尝试点可关闭按钮（`dismissBtn`：确定/关闭/X/我知道了等）自救；连续 `Prefs.unknownLimit`（默认 5，用户可调）仍 UNKNOWN 则：toast + 系统通知栏 Notification（`notifyStop`）+ 停止循环 + 等待人工重新点"开始"。

Android 13+ 首次进 `MainActivity` 申请 `POST_NOTIFICATIONS`，未授权也不影响主流程（通知静默丢弃）。

### 5.6 题库展开查看 + 搜索 + 手动修正（2026-06-26 对齐竞品题库 tab）

`MainActivity` 题库管理界面，每个题库点击「展开」后（仿竞品「搜索 / 最近十条 / 修改」）：
- **搜索框**：空 = 显示**最近入库 10 条**（题库可能几百条，不一股脑列全部）；输入关键字 = 在题干+答案里模糊搜索（按空格拆词，每词都需包含，忽略大小写，等价竞品 SQLite `LIKE`），最多显示最近命中的 50 条。
- **条目行**：题干预览 + 答案，点一下弹「修正题目」对话框，可改**题干（多行）+ 答案**或删除。
- 「最近入库」依赖插入顺序：`AnswerStore.answers` 用 `LinkedHashMap`、`persist` 按序写、`loadEntries` 按文件序读回，取末尾 N 条即最近学到的。

纯算法匹配天生不牢（阈值高了漏并、低了误并），人工核查修正是必要兜底——改完即 ground truth。`AnswerStore` 提供 `loadEntries`/`saveEntry`/`editEntry`（改题干=换 key，删旧插新）/`deleteEntry` 接口。

### 5.7 视频自动挂课（界面结构见 2.2）

`Prefs.videoMode` 开关 + `VideoMachine`（2026-06-30 从服务类拆出，见 5.9）。独立状态机，就在一个视频任务详情页内循环它的视频列表——看完一个点下一个（用户确认：不跨任务）：

- `DETAIL`：观看进度 ≥95% 则停；否则点 `开始观看` 进播放器。
- `PLAYER`：跟踪 `currentVideoIdx`（当前在看第几行）；`firstPendingIdx` = 第一个"已学 < 总长−5s"的行。该行 ≠ 当前 → 点那行播它（看完一个点下一个）、重置位置；正在看的这行位置（`posSec`）没推进=暂停/卡 → 点中间恢复播放，否则让它放；列表全看完 或 观看进度 ≥95% → 停。
- `OTHER`（疑似"下一集/继续"弹窗）→ 点 取消/关闭/知道了，自己控制进度。
- 保活：悬浮球加 `FLAG_KEEP_SCREEN_ON` 防息屏；3s 慢节奏轮询。

**2026-07-03 真机验证：主路径已通，但"完成即停"边界与"控件条自动隐藏"未验（不是完整端到端）**。用"病人参与病人安全的教育"这个 0% 未完成视频任务走了 开始→自动切集 一段，修了 2 个 bug：
- ✅ ① **每行「累计时长」边播实时累加**（列表那列秒秒在涨），可靠；观看进度% 也随之涨。判完成继续用"每行已学≥总长−5s"（累计会 replay 超总长，反而更好判）。
- ✅ ② **一集播完 → 自动切下一集**：靠 `累计≥总长` 触发 `firstPendingIdx` 前移→点下一行，真机实测 视频1(累计11:49>10:57)→自动点第2个播 8.2，无弹窗。（只见到 1→2；2→3、以及"全看完"未观察。）
- ✅ ③ **点列表行能直接切到该视频播**：真机实测点第2行→播放器立即切到视频2（`00:03 / 14:52`）、该行变粉选中、原视频累计保留。行 tap 用屏幕水平中心×行 Y。
- **修的 2 个 bug**（`NodeParser.toVideoModel`）：(a) `watchPct` 取全页最大% → 把"完成条件…95%"当已看进度→秒停，改为只从"观看进度"叶取；(b) 视频行以 `.contains(".mp4")` 为锚 → 真机名称无 .mp4 → rows 恒空 → 既不能换集也不能判完成，改为以纯序号叶 `^\d+\.$` 为锚（详见 2.2）。
- ✅ **完成判定重做（2026-07-03 二轮，关键修正）**：原 `rowDone` 用 `累计时长 ≥ 总长` 判"看完"——**大错**。累计是可 replay 累加的播放时长、和真实覆盖无关：真机一个任务三集累计全超总长，真实覆盖却是 59%/84%/100%、整体卡 83%，于是启动即 `STOP: 所有视频已看完` 在 83% **假完成误停**、永远到不了 95%。
  - **真信号 = 每条视频的绿色进度条填充比（唯一覆盖率）**。进度条无文字、`leavesOf` 收不到，新增 `allRects` 全节点 bounds 遍历 + `barCoverage()`：按几何找左≈88 的宽 View 簇，容器右=最大右、填充右=最小右，覆盖=(填充右−左)/(容器右−左)。真机读数与画面吻合(59%=421/709…、100%=797/797)。
  - `VideoMachine` 改：`rowDone = coverage ≥ 0.98`；`firstPendingIdx` 选第一条覆盖<98% 的去补；**主完成信号 = 所有绿条填满**（全≥98% ⇒ 整体≥95%，加权平均充分条件），`watchPct≥95` 仅作补充早停。
  - ⚠️ **另一真机坑：App 的"观看进度/完成进度%"是缓存死值**——播放器页播放中一直不动(实测全程 83.0776%)、连详情页都要手动点刷新 ↻ 才从 83% 跳到 100%。所以**播放中绝不能靠这个%判完成**，只能靠实时的绿条。

- ✅ **控件条自动隐藏——此播放器不隐藏，穴不成立**：真机连播 ~30s，控件条上的 `1.0x`/`640P`/`exo_content_frame` 叶始终在节点树里（是常驻自定义条、非会自动隐藏的 ExoPlayer 原生控制器），故 `isPlayer` 全程为真、正常播放**不会**误落 `OTHER`。之前 seek 时看到的 `OTHER→关弹窗`(@721,1436) 只是"视频加载中"瞬时过渡，几秒后自恢复。若日后换了会自动隐藏的播放器再考虑给 PLAYER 判定加 `exo_content_frame` 兜底。

### 5.8 考试模式（错峰 + 只显示答案，2026-07-02 重设计）

**背景**：护理助手考试页（WebView/H5）进场时（`onLoadEnd`）检测无障碍是否开启（机制见 2.3）。**一刀切**：开着任何无障碍就拦。原来"自动答题式"考试模式（`ExamMachine` 自动点选项+交卷）行不通——自动答题必须开无障碍，一进场就被拦。故重设计为**错峰 + 只显示答案、绝不自动点**：进场前关无障碍骗过检测，进场后再开、只把答案显示在浮窗上、用户自己点。

**关键约束**：无障碍一关，`DumpAccessibilityService` 就被系统销毁，它持有的 `TYPE_ACCESSIBILITY_OVERLAY` 悬浮窗随之消失。所以考试浮窗**不能**由无障碍服务托管——另起一个**前台 Service `ExamOverlayService`**，用 `TYPE_APPLICATION_OVERLAY`（需 `SYSTEM_ALERT_WINDOW` 权限）托管，独立于无障碍存活。

**流程**（`Prefs.examMode` 开关 + `ExamOverlayService` + 只读的 `ExamMachine`）：

1. **开启考试模式**（设置页开关，MainActivity）→ 启动 `ExamOverlayService`：弹 `TYPE_APPLICATION_OVERLAY` 小浮窗显示占位 `。。。`（样式同悬浮答案：浅灰小字、无底色）+ **自动关闭无障碍**（`A11yEnabler.disable`）。此时用户切到护理助手、走签到进考试页——`onLoadEnd` 检测到无障碍已关，H5 放行。
2. **双击 `。。。` 浮窗** → `ExamOverlayService` **自动开启无障碍**（`A11yEnabler.enable`）+ 浮窗文字变 `开始`。
3. 无障碍服务重新连上（`onServiceConnected`）→ 见 `examMode` 开 → **自动启动只读循环**（不需要点悬浮球▶，双击=开始信号）。`ExamMachine` 每拍读考试页题目 → **跨所有题库搜索**答案（`AnswerStore.searchAllBanks`：考试题库可能没单独刷过，故不限当前库）→ 没命中调 AI → 仍没有则 `unknown` → **广播 `EXAM_ANSWER` 把答案原文发给 `ExamOverlayService`**，浮窗更新显示。**全程不 `tap`、不 `store.put`、不建库**。
4. **音量+ 键** 隐藏/显示浮窗（无障碍服务的 `onKeyEvent` 拦 `KEYCODE_VOLUME_UP`，考试模式下广播 `EXAM_TOGGLE`，不再是开始/停止）。
5. **关闭考试模式** → 停 `ExamOverlayService`：移除浮窗 + **恢复开启无障碍**（供普通模式用）。

- 组件通信走广播（同进程，沿用本项目 START/STOP/MODE 广播风格）：`EXAM_ANSWER`(text) / `EXAM_TOGGLE`，无障碍服务→`ExamOverlayService`。
- **能骗过进场检测**（RN 事件驱动、不定时查，见 2.3）；**残余命门**：开考后 H5 若自己轮询复查会抓到再开的无障碍——APK 反不出，需真机进一次考试观察。另一命门：考试 WebView 的题目**无障碍能否读到**（H5 内容通常能读，但不保证），也需真机 dump 一次验证。这两点未验证前，考试模式属**实验性**。
- 与正常练习互不影响（独立开关、独立浮窗服务、只读不点）。

### 5.9 状态机分层 + 纯函数单测（2026-06-30；QuizMachine 抽取 2026-07-01）

- **状态机分层**：`DumpAccessibilityService` 原是 821 行上帝类、四套状态机混在一起。已按模式**全部拆分**——答题 `QuizMachine` / 悬浮 `FloatMachine` / 视频 `VideoMachine` / 考试 `ExamMachine` 各自独立成类、持有私有战术状态（题干/失败计数/过渡去重/提交冷却等），通过 `AutoHost` 接口（Service 实现）取用 `tap/tapSequence/store/overlay/scheduleNextStep/stopAuto` 等公共能力；`loopStep` 改用 `Prefs.mode()`→`AnswerMode` 枚举分派（行为与原 if 顺序一致）。**组进度（`groupsDone`/`answered`/`targetGroups`）是跨切面状态**（悬浮球显示 + 停止汇总 + 考试共用 answered），仍留 Service 并经 `AutoHost` 暴露给答题机读写。抽取逐行等价照搬，答题主路径逻辑未改。Service 由 821 行降到约 475 行，答题逻辑进 `QuizMachine`（约 210 行）。
- **悬浮球不再闪**：原每步 `setVisible(false/true)` 藏球（OCR 截图时代"先藏球再截图"的遗留）触发窗口重排=闪烁；控件直读不截图、球在边缘与点击（屏幕中部）不重叠，已去掉每步藏球，`setVisible` 仅暂停/停止用，读题同步即得更快。
- **纯函数单测**：模糊匹配抽到 `TextMatch`、答案字母↔下标↔文字抽到 `AnswerCodec`、`NodeParser` 的时间/字母解析放宽可见性——三者均有 JUnit 单测（`app/src/test`，27 例，`./gradlew :app:testDebugUnitTest`），不需真机即可跑，钉住 0.82/0.84 等关键阈值与解析边界。

## 7. 竞品对齐：现状对账 + 待办路线图（2026-06-26 讨论）

> 对照竞品笔记 `apk_analysis/notes/竞品亮点速记.md`（11 节）。竞品 = AutoJs Pro 套壳「小助手」，目标同为护理助手。

### 7.1 已对齐（功能+逻辑一致）

控件树读题 · 自动开无障碍（`A11yEnabler`）· 服务名伪装 · 题干归一化（治 key 漂移）· 盲选建库（一边刷一边建）· 答错自动修正题库（反馈页覆盖自愈）· 两种模式 win/win2 · **悬浮答案只显示原文**（5.2）· **题库管理：模糊搜索 / 最近 10 条 / 增删改修正**（5.6）· 单选/多选/判断题型 · 组目标 · DB 健壮性。
